---
title: "解决复杂问题"
description: 使用问题解决工作流诊断根因并找到解决方案
---

使用 `problem-solving` 工作流，系统性地诊断问题、识别根因，并运用经过验证的分析框架制定有效的解决方案。

## 什么时候使用

- 面对复杂、持续存在的挑战
- 需要在修复表象之前理解根因
- 存在多个解决方案选项需要选择
- 问题反复出现，尽管之前已经尝试修复
- 问题跨越多个系统或组件

## 什么时候跳过

- 已充分理解、解决方案显而易见的问题
- 简单、孤立的 bug 或问题
- 时间紧迫、需要立即行动的情况

:::note[前置条件]
- **已安装带 CIS 的 BMad Method** — Dr. Quinn 是 Creative Intelligence Suite 的一部分
- **一个明确的问题** — 你想要解决的东西
- **对分析的耐心** — 根因探索需要时间
:::

## 步骤

### 1. 加载 Dr. Quinn

开始新对话并加载 Problem Solver：

```
/cis-problem-solving
```

### 2. 描述你的问题

Dr. Quinn 会问你的挑战。对症状要具体：

**好的问题描述：**
- "用户在输入支付信息后 40% 的概率放弃结账"
- "当用户数超过 1000 时，数据库查询需要 30 秒"
- "上次发版后客户支持工单增加了 300%"

**效果较差：**
- "修复性能"（太模糊）
- "应用很慢"（没有具体信息）

### 3. 诊断问题

Dr. Quinn 把问题当作谜题来对待，引导你完成：

| 诊断步骤 | 目标 |
| -------- | ---- |
| **症状分析** | 区分你看到的和导致它的原因 |
| **边界定义** | 理解问题存在和不存在的范围 |
| **历史回顾** | 什么改变了、什么时候、尝试过什么 |

### 4. 运用分析框架

Dr. Quinn 从他的 solving-methods 库中选择：

| 框架 | 适合场景 |
| ---- | -------- |
| **五个为什么** | 快速根因追溯 |
| **TRIZ** | 技术矛盾和发明性解决方案 |
| **约束理论** | 系统瓶颈和流程 |
| **系统思维** | 相互关联的反复出现的问题 |
| **根因分析** | 全面的因果关系映射 |

:::tip[Dr. Quinn 的风格]
Dr. Quinn 像福尔摩斯和俏皮科学家的结合——演绎、好奇，突破时刻会喊出"啊哈！"
:::

### 5. 生成和评估解决方案

一旦根因被识别：

1. **发散** — 生成多种解决方案
2. **评估** — 评估优缺点和权衡
3. **选择** — 基于有效性和可行性做出选择
4. **规划** — 创建附带风险缓解的实施步骤

## 你将获得

输出保存到 `_bmad-output/problem-solution-{date}.md`：

| 部分 | 内容 |
| ---- | ---- |
| **问题陈述** | 清晰定义的挑战及症状 |
| **诊断** | 使用所选框架的根因分析 |
| **解决方案选项** | 多种方案及优缺点 |
| **推荐方案** | 最佳选项及理由 |
| **实施计划** | 可执行的步骤和时间线 |
| **风险缓解** | 可能出错的情况及预防措施 |
| **成功指标** | 如何衡量有效性 |

## 示例

```text
你：/cis-problem-solving
Dr. Quinn：今天我们在解什么谜题？
你：上次发版后用户参与度下降了 50%。
Dr. Quinn：迷人的症状。让我们深入挖掘。
       [运用五个为什么]
       为什么参与度下降了？
你：用户不再打开应用了。
Dr. Quinn：他们为什么不打开了？
你：推送通知停止工作了。
Dr. Quinn：通知为什么停止了？
你：API 变了，我们没有更新。
Dr. Quinn：啊哈！根因：API 集成缺失。
       但为什么这个问题没有在测试中被发现？
       [更深入地挖掘流程问题]
       [识别多个根因]
       [生成解决方案选项]
       [创建附带安全措施的实施计划]
```

## 技巧

:::tip[最佳实践]
- **区分症状和原因** — 你看到的不是问题本身
- **质疑假设** — "显而易见"的原因往往是错的
- **系统性思考** — 问题通常有多个相互关联的原因
:::

:::caution[避免常见错误]
- 不要急着跳到解决方案 — Dr. Quinn 会让你慢下来
- 不要接受第一个答案 — 继续深挖
- 不要忽略人的因素 — 流程和人的问题会导致技术症状
:::

## 下一步

问题解决之后：

- 使用**头脑风暴**（`/cis-brainstorm`）生成创意解决方案
- 运用**创新策略**（`/cis-innovation-strategy`）如果问题需要战略转型
- 使用**设计思维**（`/cis-design-thinking`）如果用户正在经历这个问题

## 提供上下文

为了获得最佳效果，通过 `--data` 标志提供问题上下文：

```bash
workflow cis-problem-solving --data /path/to/problem-brief.md
```

Dr. Quinn 会使用这些上下文加速诊断阶段。
